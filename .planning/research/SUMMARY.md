# Research Summary: HipsterStyle Article System Rebuild

**Domain:** WordPress-safe article HTML templates with N8N workflow automation
**Researched:** 2026-03-25
**Overall confidence:** HIGH

## Executive Summary

The HipsterStyle Article System Rebuild operates under a single dominant constraint: WordPress's `wp_kses_post()` HTML sanitizer strips `<style>` blocks, external stylesheets, and most `<script>` tags from post content. Every technology and design decision must flow from this constraint. The existing baseline template (in `claude-code/`) already demonstrates the correct approach -- inline CSS on every element, `<details>`/`<summary>` for accordions, `onmouseover`/`onmouseout` for hover states -- but still includes a `<style>` block for hover transitions and class-based selectors that WordPress would strip in production.

The N8N workflow pipeline is well-architected in the existing codebase. The `[SQUARE_BRACKET]` placeholder system correctly avoids N8N's `{{ expression }}` evaluation. Firecrawl integration for site discovery and product extraction is already implemented in the improved workflow JSON. The main gap is that Firecrawl product extraction for hipsterstyle.co.il specifically has not been validated -- the site's product page structure needs verification during implementation.

Supabase Storage with public buckets is the correct choice for image hosting. The existing codebase already has Supabase client integration. Product images from external CDNs (Wix, Shopify) should be mirrored to Supabase to prevent URL breakage. The public bucket approach eliminates auth complexity and maximizes CDN cache hit rates.

The rebuild is fundamentally a **constraint-driven HTML/CSS engineering problem**, not a framework selection problem. There is no framework to choose. The output is a single HTML fragment with inline styles that must survive WordPress rendering, look professional on mobile/tablet/desktop, support Hebrew RTL, and be generated programmatically by an LLM via N8N.

## Key Findings

**Stack:** Pure inline CSS HTML template generated by LLM via N8N, with Firecrawl product discovery and Supabase image hosting. No frameworks, no dependencies, no build step.

**Architecture:** Linear N8N pipeline: Google Docs trigger -> Parse -> Firecrawl site discovery -> Firecrawl product scrape (conditional) -> Supabase image upload (conditional) -> LLM article generation -> WordPress safety post-processing -> WP REST API publish.

**Critical pitfall:** The existing template contains a `<style>` block (lines 2-18 of `claude-code/Improved_HTML_Template.html`) that WordPress WILL strip. All hover transitions defined there (`om-icon`, `om-toc-link`, `om-card`, `om-float-btn`, `om-cta`) will silently disappear in production. This is the #1 issue to fix.

## Implications for Roadmap

Based on research, suggested phase structure:

1. **Audit & WordPress Rendering Risk Assessment** - Rationale: Must understand exactly what breaks before redesigning. The existing template has known WordPress-unsafe patterns.
   - Addresses: WordPress rendering risk review, full audit of baseline files
   - Avoids: Building on a broken foundation

2. **Firecrawl Product Discovery & Verification** - Rationale: Product data feeds into the template. Must verify Firecrawl can extract hipsterstyle.co.il product pages before designing product card layouts.
   - Addresses: Firecrawl product discovery from inner pages, social profile verification
   - Avoids: Designing for data that cannot be reliably extracted

3. **WordPress-Safe HTML Template Rebuild** - Rationale: Core deliverable. All inline CSS, zero `<style>` blocks, RTL support, responsive via `clamp()`. This is the longest phase.
   - Addresses: TOC, product cards, FAQ accordion, floating buttons, author section, hover states, responsive QA
   - Avoids: The `<style>` block pitfall, class-based selectors, media queries

4. **N8N Prompt & Workflow Alignment** - Rationale: Prompt must match the new template structure exactly. Workflow JSON must integrate Firecrawl/Supabase nodes correctly.
   - Addresses: N8N prompt with exact injection block, workflow JSON updated, Supabase image hosting
   - Avoids: Prompt/template mismatch, N8N expression evaluation errors

5. **QA Validation & WordPress Deployment** - Rationale: Final validation in live WordPress environment. Mobile/tablet/desktop responsive testing.
   - Addresses: Mobile/tablet/desktop responsive QA, WordPress rendering validation
   - Avoids: Shipping untested code to production

**Phase ordering rationale:**
- Phase 1 before 3 because audit findings drive template requirements
- Phase 2 before 3 because product data structure determines card layout implementation
- Phase 3 before 4 because the prompt must describe the exact template structure
- Phase 4 before 5 because workflow must produce valid output before QA testing

**Research flags for phases:**
- Phase 2: Needs implementation-time research (Firecrawl extraction accuracy for hipsterstyle.co.il specifically)
- Phase 3: Standard patterns, well-understood constraints. Unlikely to need additional research.
- Phase 5: May need research if WordPress theme CSS conflicts with inline styles

## Gaps to Address

- Exact `$allowedposttags` list for the target WordPress installation (mahsan.websreport.net) -- may have custom filters from theme/plugins
- Whether the WordPress admin user publishing via N8N API bypasses wp_kses entirely (likely yes for admin role, but should verify)
- Firecrawl extraction accuracy for hipsterstyle.co.il product pages -- needs live testing
- Whether Arimo font is loaded by the WordPress theme on mahsan.websreport.net -- if not, articles fall back to Arial (acceptable but less polished)
- `position: fixed` and `z-index` behavior within the WordPress theme's CSS context (floating buttons may conflict with theme elements)
